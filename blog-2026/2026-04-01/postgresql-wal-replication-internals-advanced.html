Deep Dive into PostgreSQL WAL Replay: Logical Decoding, Replication Slots, and Production Scaling

PostgreSQL guarantees durability and point‑in‑time recovery through its Write‑Ahead Logging (WAL) subsystem. Every change to a data page is first recorded in the WAL before the page itself is flushed to disk. This design decouples transaction commit latency from physical I/O, enables crash recovery, and forms the backbone of logical replication.

When a logical subscriber connects, the server streams a filtered view of the WAL that contains only row‑level changes. The logical decoding pipeline transforms raw WAL records into a stream of change sets that can be replayed on a downstream system. Replication slots preserve the minimum required WAL position, preventing premature recycling and ensuring that all subscribers stay in sync.

The following sections dissect the pipeline, examine the slot lifecycle, and explore scaling considerations that arise in production environments.

Logical Decoding Pipeline

The logical decoding process is orchestrated by three core components:

  1. walreceiver – runs on the subscriber, establishes a replication connection, and pulls WAL data from the primary.
  2. waldecoder – part of the walsender on the primary, it applies output plugins (e.g., pgoutput, wal2json) to translate raw WAL into logical messages.
  3. apply worker – on the subscriber, consumes the logical stream and replays changes inside a transaction.

The flow can be visualised as:

primary (walsender) → network → subscriber (walreceiver) → decoder → apply worker

Each logical message carries a Log Sequence Number (LSN) that marks its position in the original WAL. The subscriber records the last applied LSN in its replication slot state, allowing the primary to truncate WAL safely.

Replication Slot Lifecycle

A replication slot is created with SELECT * FROM pg_create_logical_replication_slot('slot_name', 'pgoutput');. The slot persists across restarts and tracks two LSNs:

  • restart_lsn – the oldest WAL byte still needed by any subscriber.
  • confirmedflushlsn – the last LSN that the subscriber has acknowledged as safely written.

When a subscriber disconnects, the slot remains active unless explicitly dropped. This prevents WAL removal but can lead to uncontrolled growth if the slot is idle for long periods.

Managing Slot Bloat

sql
SELECT slotname, restartlsn, pgsizepretty(pgwallsndiff(pgcurrentwallsn(), restartlsn)) AS retained FROM pgreplication_slots;

Regular monitoring and automated cleanup (e.g., pg_drop_replication_slot) are essential in large clusters.

LSN Tracking and Snapshot Management

Logical decoding must present a consistent view of the database at each transaction boundary. The decoder captures a snapshot identifier (snapshot_name) that the subscriber can reuse when applying changes out of order. Internally, PostgreSQL stores these snapshots in pg_snapshots and ties them to the LSN at which the snapshot was taken.

When a subscriber requests a restart from a specific LSN, the primary validates that the LSN is not older than restart_lsn. If it is, the subscriber must perform a base backup to re‑initialize.

WAL Receiver / Sender and Parallel Apply Workers

The walreceiver process runs under the replication role and writes incoming WAL chunks to a temporary file (walreceiver.pid). The walsender on the primary respects the wal_sender_timeout and wal_receiver_status_interval settings to avoid stalls.

From PostgreSQL 14 onward, logical replication can leverage parallel apply workers. Setting max_logical_replication_workers and max_sync_workers_per_subscription enables the subscriber to split the incoming stream into multiple apply threads, each handling a subset of tables. This reduces latency for high‑throughput workloads.

conf

postgresql.conf (subscriber)

maxlogicalreplicationworkers = 8 maxsyncworkerspersubscription = 4 walreceiverstatusinterval = 10s

Performance and Scaling

Checkpoint Tuning

Checkpoints determine when dirty buffers are flushed to disk. A too‑aggressive checkpoint interval (checkpoint_timeout) can cause frequent WAL file recycling, increasing I/O pressure on the primary. Conversely, a very long interval may delay WAL removal, inflating disk usage for active slots.

conf

Example balanced settings

checkpointtimeout = 15min maxwalsize = 4GB minwal_size = 1GB

wal_buffers Sizing

wal_buffers reserves shared memory for WAL data before it is written to the WAL segment. For logical replication with many concurrent subscribers, increasing this buffer reduces back‑pressure on the walsender.

conf
walbuffers = 16MB # default is 1/64 of sharedbuffers

Network Back‑Pressure

The primary monitors the subscriber's feedback via StandbyStatusUpdate messages. If the subscriber lags, the primary throttles WAL transmission, which can be observed in pg_stat_replication under write_lag and flush_lag. Adjusting wal_sender_timeout and wal_receiver_timeout helps avoid premature disconnections.

Scaling to Many Subscribers

When the number of logical subscribers grows, the primary's CPU usage for decoding can become a bottleneck. Strategies include:

  • Deploying a dedicated decoding node that receives WAL once and forwards decoded messages to multiple downstream consumers.
  • Using partitioned publications to limit the volume of data each subscriber receives.
  • Leveraging pglogical extensions that support multi‑master topologies and more efficient change capture.

Common Failure Modes

Replication Slot Bloat

If a slot is idle, restart_lsn stops advancing, causing WAL files to accumulate. Disk exhaustion is a frequent symptom. Mitigation steps:

  1. Monitor pg_replication_slots regularly.
  2. Set wal_keep_size as a safety net.
  3. Automate slot retirement after a configurable idle period.

Mis‑aligned LSN and Data Loss

Attempting to restart a subscriber from an LSN older than restart_lsn results in an error:

ERROR: could not find a valid restart point for replication slot "slot_name"

The only safe recovery is to take a fresh base backup and recreate the subscription. To avoid this, ensure that subscribers acknowledge progress promptly (pg_replication_origin_advance).

Inconsistent Snapshots

When a subscriber applies changes out of order, foreign key constraints may temporarily fail. Logical decoding plugins provide the include-timestamp and include-transaction options to help reconstruct the original order. If ordering cannot be guaranteed, consider disabling constraint checks during bulk apply and re‑enabling them afterward.

Key Takeaways

  • WAL is the single source of truth; logical decoding translates it into a consumable change stream.

  • Replication slots protect required WAL but must be actively managed to prevent bloat.

  • Proper LSN tracking, snapshot handling, and feedback loops are essential for reliable replay.

  • Performance hinges on checkpoint configuration, wal_buffers, and network latency; parallel apply workers can scale out replay work.

  • Common pitfalls include idle slots causing disk exhaustion and attempts to restart from stale LSNs, both of which demand proactive monitoring and clear recovery procedures.
