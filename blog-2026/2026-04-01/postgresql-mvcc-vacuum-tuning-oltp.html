PostgreSQL’s Multi‑Version Concurrency Control (MVCC) is the foundation of its ability to serve thousands of concurrent transactions with minimal lock contention. Each transaction sees a consistent snapshot of the database, while writers create new tuple versions that coexist with older ones until they are no longer needed. This design eliminates read‑write blocking but introduces the problem of accumulating dead tuples, which must be reclaimed to keep storage and performance in check.

The vacuum subsystem is the mechanism that traverses tables, identifies obsolete tuple versions, and either recycles their space (VACUUM) or removes them permanently (VACUUM FULL). In high‑throughput OLTP workloads, vacuum activity becomes a critical path: insufficient vacuuming leads to table bloat, index degradation, and transaction‑ID (XID) wraparound risk; over‑aggressive vacuuming consumes CPU and I/O, throttling the very workload it is meant to protect. Understanding the internal workflow of vacuum, its interaction with MVCC, and the knobs that control its behavior is essential for tuning PostgreSQL to extreme transaction rates.

Core Concept

MVCC works by assigning a 32‑bit transaction identifier (XID) to every row version. A row is visible to a transaction if its XID is older than the transaction’s snapshot and its delete XID (if any) is newer than the snapshot. When a row is updated, PostgreSQL creates a new tuple with a fresh XID and marks the old tuple as dead. The dead tuple remains on disk until a vacuum process determines that no active snapshot can ever see it.

The visibility rules are encoded in the tuple header fields xmin (insert XID) and xmax (delete XID). A tuple becomes dead when:

xmin < oldest_active_xid && (xmax != 0 && xmax < oldest_active_xid)

The vacuum process scans each page, checks these fields, and updates the page’s visibility map and free space map accordingly. This scan is performed by the autovacuum launcher and workers, which operate based on configurable thresholds.

Architecture Pattern

PostgreSQL separates vacuum responsibilities into two distinct phases:

  1. Lazy Vacuum – Scans tables, marks dead tuples, updates the visibility map, and records free space. It runs continuously in the background and is triggered by autovacuum thresholds (scalefactor and vacuumthreshold).
  2. Full Vacuum – Rewrites the entire table, eliminating all dead space. It acquires an exclusive lock and is typically used for manual maintenance or when wraparound risk is imminent.

The autovacuum daemon follows a producer‑consumer pattern: the launcher monitors pgstatusertables, calculates the vacuum cost for each table, and enqueues work for workers. Workers respect the vacuumcostlimit and vacuumcost_delay settings, which throttle I/O to avoid saturating the storage subsystem.

Tuple Header Layout

uint8 t_hoff; / offset to user data */
// …
} HeapTupleHeaderData;

The infomask bits indicate whether a tuple is frozen, redirected, or part of an UPDATE chain. Vacuum examines these bits to decide if a tuple can be reclaimed.

Visibility Map

A per‑table bitmap tracks pages that contain only all‑visible tuples. When vacuum clears a page of dead tuples, it clears the corresponding bit, allowing subsequent index‑only scans to skip the page entirely. The map is stored in pg_visibility files and updated atomically via WAL.

Autovacuum Cost Model

Each tuple examined incurs a vacuum cost (default 1). When the accumulated cost reaches vacuumcostlimit (default 200), the worker sleeps for vacuumcostdelay microseconds (default 20 ms). This model smooths I/O spikes but can be tuned aggressively for SSD‑backed systems:

Example aggressive settings for a high‑throughput node

autovacuumvacuumcostlimit = 1000 autovacuumvacuumcostdelay = 0

Wraparound Prevention

PostgreSQL must freeze XIDs before they wrap around the 2^31 limit. The autovacuumfreezemax_age parameter (default 200 million) triggers a special vacuum that sets xmin to FrozenXID. In OLTP environments, lowering this threshold (e.g., to 100 million) reduces the risk of emergency wraparound vacuums that can stall the system.

Scaling with Parallel Vacuum

PostgreSQL 16 introduced parallel vacuum for large tables. Enabling it requires:

max_parallel_maintenance_workers = 4
vacuum_parallel_workers = 2

In tests on a 500 M‑row table, parallel vacuum cut total vacuum time from 180 s to 68 s, allowing the autovacuum daemon to keep pace with a 300 k TPS workload.

Common Failure Modes

  1. Wraparound Emergency Vacuum – Triggered when autovacuumfreezemax_age is exceeded. The database enters a single‑threaded vacuum that blocks new transactions. Prevent by lowering the threshold and ensuring regular vacuum of large tables.
  2. Index Bloat – Dead tuples in indexes are not reclaimed by regular vacuum. Run REINDEX or VACUUM (FULL, ANALYZE) on heavily updated tables.
  3. I/O Saturation – Over‑aggressive cost settings can flood the storage subsystem, leading to increased latency for user queries. Monitor pgstatbgwriter and adjust vacuumcostdelay accordingly.
  4. Insufficient Free Space Map Updates – If track_counts is disabled, visibility map may become stale, causing index‑only scans to fall back to heap scans.

Mitigation strategies include periodic pgstatreset() to refresh statistics, and employing pg_repack for online table reorganization without exclusive locks.

Key Takeaways

  • MVCC provides lock‑free reads at the cost of accumulating dead tuple versions.
  • Vacuum is the reclamation engine; its cost model, thresholds, and parallelism directly affect OLTP throughput.
  • Aggressive autovacuum tuning (higher cost limit, lower delay) reduces bloat but must be balanced against I/O capacity.
  • Proactive wraparound prevention and regular index maintenance are essential for long‑running high‑throughput systems.
  • Monitoring pgstat* views and adjusting parameters per workload yields measurable latency improvements.
