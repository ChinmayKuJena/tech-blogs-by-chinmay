Idempotent Writes with PostgreSQL UPSERT (ON CONFLICT)

📅 2026-04-07
Learn how PostgreSQL's ON CONFLICT clause makes write operations idempotent, eliminating duplicate rows and race conditions without extra application logic.
postgresql sql database idempotency

Insight

When an API endpoint must tolerate retries (e.g., network glitches or client retries), naïve INSERT statements can create duplicate rows or raise errors. PostgreSQL’s ON CONFLICT clause turns a plain INSERT into an upsert: it either inserts a new row or updates the existing one based on a unique constraint. This single statement is atomic, sidestepping race conditions that plague a two‑step "check‑then‑insert" approach and keeping your data model clean.

Example

-- Assume a unique index on (user_id, order_id)
INSERT INTO orders (user_id, order_id, status, created_at)
VALUES (42, 1001, 'pending', NOW())
ON CONFLICT (user_id, order_id) DO UPDATE
  SET status = EXCLUDED.status,      -- keep the latest status
      created_at = GREATEST(orders.created_at, EXCLUDED.created_at);

The EXCLUDED alias holds the values that would have been inserted, allowing selective column updates.

Takeaway

Use ON CONFLICT for any write that must be safe to retry. Define proper unique constraints, and let PostgreSQL handle the idempotency – you’ll reduce code complexity and eliminate subtle race bugs.
