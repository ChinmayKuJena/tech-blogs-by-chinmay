Optimistic Concurrency Control with JPA @Version

📅 2026-03-24
Learn how a simple @Version field in JPA entities provides safe optimistic locking without heavy infrastructure, preventing lost updates in high‑traffic services.
java spring boot database optimistic locking

Insight

In many legacy systems developers reach for pessimistic locks or external version services to avoid race conditions. The myth is that optimistic locking requires complex tooling; in reality, JPA’s @Version annotation adds a lightweight version column that the persistence provider checks on every UPDATE. If two requests modify the same row concurrently, the second transaction fails with an OptimisticLockException, letting you retry or reject the operation. This approach scales well because it avoids row‑level locks and works transparently with connection pools.

Example

@Entity
public class Order {
    @Id @GeneratedValue Long id;
    private String status;
    @Version
    private Long version; // auto‑incremented on each update
}

// Service method
@Transactional
public void shipOrder(Long orderId) {
    Order order = repo.findById(orderId).orElseThrow();
    order.setStatus("SHIPPED");
    // commit triggers version check; throws OptimisticLockException on conflict
}

Takeaway

Add a @Version field to any entity that experiences concurrent writes. Handle OptimisticLockException by retrying the transaction or returning a clear error to the client. This gives you safe concurrency control with virtually no performance penalty.
