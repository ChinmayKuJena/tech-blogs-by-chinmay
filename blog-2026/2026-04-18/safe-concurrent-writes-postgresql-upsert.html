Safe Concurrent Writes with PostgreSQL UPSERT

📅 2026-04-18
Learn how PostgreSQL's ON CONFLICT clause can turn race‑prone inserts into atomic upserts, avoiding duplicate‑key errors and keeping data consistent.
postgresql sql backend database

Insight

When multiple workers try to insert the same logical record, classic INSERT statements race and cause duplicate‑key errors. PostgreSQL's ON CONFLICT … DO UPDATE (often called upsert) makes the operation atomic: the engine either inserts the row or updates the existing one in a single statement, eliminating the need for explicit locks or retry loops. However, beware of silent overwrites—only update columns that truly need refreshing, and consider using the EXCLUDED pseudo‑table to reference incoming values.

Example

INSERT INTO inventory (product_id, quantity, updated_at)
VALUES (42, 10, now())
ON CONFLICT (product_id) DO UPDATE SET
  quantity = inventory.quantity + EXCLUDED.quantity,
  updated_at = EXCLUDED.updated_at;

This query increments the stock for product_id = 42 safely, even if dozens of workers run it simultaneously.

Takeaway

Prefer INSERT … ON CONFLICT for idempotent writes. Keep the DO UPDATE clause minimal to avoid unintended data loss, and always index the conflict target column for optimal performance.
