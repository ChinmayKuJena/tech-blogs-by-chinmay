Optimistic vs Pessimistic Locking: When to Use Each

📅 2026-03-21
A quick guide to choosing between optimistic and pessimistic locking in relational databases, with a tiny TypeORM example.
databases postgresql locking performance

Insight

When many users update the same row, race conditions can corrupt data. Optimistic locking assumes conflicts are rare: it stores a version column and aborts the transaction if the version changed. This adds virtually no overhead in low‑contention scenarios and scales well with read‑heavy workloads. Pessimistic locking acquires a row‑level lock (SELECT … FOR UPDATE) at the start of the transaction, guaranteeing exclusive access but increasing lock contention and blocking other queries. Use optimistic locking for APIs that mostly read and rarely clash, and switch to pessimistic locking for critical sections like inventory decrements or financial transfers where lost updates are unacceptable.

Example

// TypeORM entity with a version column for optimistic locking
@Entity()
export class Account {
  @PrimaryGeneratedColumn()
  id: number;

  @Column('decimal')
  balance: number;

  @VersionColumn()
  version: number; // auto‑incremented on each UPDATE
}

// Optimistic update
await getRepository(Account).update({ id, version }, { balance: newBal });

// Pessimistic update (PostgreSQL)
await getManager().transaction(async tx => {
  const acc = await tx.findOne(Account, id, { lock: { mode: 'pessimistic_write' } });
  acc.balance = newBal;
  await tx.save(acc);
});

Takeaway

Start with optimistic locking—it's cheap and safe for most web APIs. Reserve pessimistic locks for hot spots where the cost of a retry outweighs the cost of blocking, and always monitor lock wait times in production.
