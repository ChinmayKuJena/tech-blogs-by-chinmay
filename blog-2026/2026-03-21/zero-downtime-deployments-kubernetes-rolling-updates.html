Zero‑Downtime Deployments with Kubernetes Rolling Updates

📅 2026-03-21
Learn how Kubernetes rolling updates keep services available during deployments and avoid traffic spikes.
kubernetes devops deployment undefined

Insight

Kubernetes rolling updates let you replace pods incrementally while maintaining the desired replica count, preventing downtime and protecting against sudden load spikes. By tuning maxSurge and maxUnavailable, you control how many extra pods are created and how many old pods can be terminated at once. This fine‑grained control is essential for stateful services where abrupt pod termination could cause request failures or data loss.

Example

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # allow one extra pod during update
      maxUnavailable: 0  # never drop below desired replicas
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
      - name: api
        image: myorg/api:{{VERSION}}
        ports:
        - containerPort: 8080

Takeaway

Set maxSurge to a small number (e.g., 1) and maxUnavailable to 0 for true zero‑downtime releases. Combine this with readiness probes so new pods only receive traffic after they’re fully healthy, ensuring seamless user experiences during continuous deployment.
