Optimistic Concurrency with Version Columns in PostgreSQL

📅 2026-04-11
Use a simple version column to prevent lost updates without heavy locking, and see a minimal TypeORM example for NestJS services.
postgresql optimistic-concurrency typeorm backend

Insight

Optimistic concurrency control (OCC) lets multiple workers read the same row and only fails when they try to write conflicting changes. By adding a tiny int version column that increments on each update, you avoid expensive SELECT ... FOR UPDATE locks while still guaranteeing data integrity. PostgreSQL’s WHERE version = :oldVersion clause turns the check into an atomic operation, and the database returns 0 rows if another transaction already modified the record.

Example

// entity.ts (TypeORM)
@Entity()
export class Account {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  balance: number;

  @VersionColumn()
  version: number; // auto‑incremented on each UPDATE
}

// service.ts
async function credit(id: number, amount: number) {
  const repo = dataSource.getRepository(Account);
  const account = await repo.findOneBy({ id });
  account.balance += amount;
  // TypeORM includes the version in the UPDATE's WHERE clause
  await repo.save(account);
}

Takeaway

Add a @VersionColumn (or a manual version field) to any table where concurrent writes are possible. The extra byte per row is negligible, and you get safe, lock‑free updates with zero code‑level race conditions.
