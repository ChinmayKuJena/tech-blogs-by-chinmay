Logical Decoding Internals: Optimizing WAL Streaming and Replication Slot Management in PostgreSQL

A deep examination of PostgreSQL's logical decoding pipeline, replication slot lifecycle, and how to tune them for high‑throughput change data capture while avoiding common pitfalls.

Logical decoding provides a programmable view of the write‑ahead log (WAL) that can be consumed by external systems for change data capture (CDC). The feature sits on top of PostgreSQL's core replication infrastructure, exposing each committed transaction as a series of logical change events. Understanding how the decoder interacts with WAL, replication slots, and the streaming protocol is essential for building reliable, high‑throughput pipelines.

This article walks through the internal architecture, the lifecycle of replication slots, practical plugin development tips, and the performance knobs that keep the system from choking under load. It also highlights common misconfigurations that lead to slot bloat, WAL retention overflow, or even data loss.

Logical Decoding Architecture and WAL Format Fundamentals

PostgreSQL writes every modification to the WAL before the data page is flushed. Logical decoding does not emit raw WAL bytes; instead it parses the WAL records, reconstructs the transaction context, and forwards a logical representation to an output plugin. The pipeline consists of three stages:

  1. WAL Reader – a background worker that streams WAL pages from the pg_wal directory.
  2. Decoder Core – transforms physical WAL records into logical change sets, handling transaction boundaries, sub‑transactions, and visibility rules.
  3. Output Plugin – a user‑supplied module that formats the logical changes (e.g., JSON, Avro) and decides when to emit them.

The logical decoding API relies on a snapshot that guarantees a consistent view of the database at the point the transaction started. By default the decoder uses a transaction snapshot (the same one used for MVCC), but plugins can request a logical snapshot that includes all changes visible to the decoding session.

Replication Slot Lifecycle and Consumer Coordination Pattern

A replication slot guarantees that the server retains WAL files until every consumer has processed them. Slots are created with pg_create_logical_replication_slot and identified by a unique name. The lifecycle follows a simple state machine:

  • Inactive – slot exists but no client is connected. WAL retention is governed by wal_keep_size and max_slot_wal_keep_size.
  • Active – a client has called START_REPLICATION and is streaming changes. The server tracks the LSN up to which the client has confirmed receipt.
  • Dropping – when pg_drop_replication_slot is issued, the slot is marked for removal; any active streaming connection must terminate first.

A typical consumer pattern is:

SELECT * FROM pg_create_logical_replication_slot('my_slot', 'pgoutput');
# Connect using libpq or a driver that supports logical replication
START_REPLICATION SLOT my_slot LOGICAL <start_lsn>;
# Process messages, periodically send feedback with the latest LSN
SELECT pg_replication_origin_advance('my_slot', <last_lsn>);

The feedback messages are crucial: they tell the server how far the consumer has progressed, allowing it to recycle old WAL segments.

Plugin Development, Output Plugins, and Snapshot Handling

Output plugins are compiled as shared libraries and loaded by the server at slot creation time. The API consists of a handful of callbacks:

  • pg_output_plugin_init – registers the plugin and its options.
  • pg_output_startup – receives the initial snapshot and replication start LSN.
  • pg_output_begin_txn, pg_output_commit_txn – delimit transaction boundaries.
  • pg_output_change – called for each row‑level change (INSERT/UPDATE/DELETE).
  • pg_output_shutdown – cleanup before the slot is dropped.

A minimal plugin skeleton (indented as a code block) looks like this:

#include "postgres.h"
#include "replication/logical.h"

void _PG_output_plugin_init(OutputPluginCallbacks *cb)
{
    cb->startup_cb = pg_output_startup;
    cb->begin_cb   = pg_output_begin_txn;
    cb->change_cb  = pg_output_change;
    cb->commit_cb  = pg_output_commit_txn;
    cb->shutdown_cb = pg_output_shutdown;
}

static void pg_output_startup(LogicalDecodingContext *ctx, OutputPluginOptions *opt, bool is_init)
{
    opt->output_type = OUTPUT_PLUGIN_OUTPUT_LOGICAL;
    /* Request a logical snapshot if needed */
    ctx->snapshot = GetTransactionSnapshot();
}

static void pg_output_change(LogicalDecodingContext *ctx,
                             ReorderBufferTXN *txn,
                             Relation rel,
                             ReorderBufferChange *change)
{
    /* Emit a simple JSON line */
    StringInfoData buf;
    initStringInfo(&buf);
    appendStringInfo(&buf, "{\"table\":\"%s\",\"op\":\"%c\",\"data\":%s}\n",
                     RelationGetRelationName(rel),
                     change->action,
                     row_to_json(change->data));
    OutputPluginWrite(ctx, buf.data, buf.len);
}

The snapshot handling strategy influences both latency and consistency. A static snapshot (taken at slot creation) ensures repeatable reads but can cause long‑running slots to retain large WAL ranges. A dynamic snapshot (requested per transaction) reduces retention but may miss intermediate changes if the consumer restarts.

Throughput Bottlenecks, Slot Bloat, and Tuning Parameters

When the decoder cannot keep up with the incoming WAL rate, the slot's LSN lags, and PostgreSQL is forced to keep WAL files longer. This manifests as slot bloat and can exhaust disk space. The primary knobs to control this behavior are:

  • max_replication_slots – total number of slots the server will allow.
  • max_wal_size / min_wal_size – bounds for automatic WAL recycling.
  • wal_keep_size – explicit amount of WAL to retain for standby or slot consumers.
  • wal_sender_timeout – how long the server waits for feedback before assuming the client is dead.
  • max_slot_wal_keep_size – per‑slot limit for retained WAL.

Typical tuning steps:

  1. Measure baseline throughput – monitor pg_stat_replication and pg_stat_wal_receiver for sent_lsn, write_lsn, and flush_lsn.
  2. Increase max_slot_wal_keep_size if the consumer processes bursts slower than the write rate.
  3. Adjust wal_sender_timeout to a lower value (e.g., 30s) to force slot cleanup when a client disappears.
  4. Enable wal_compression to reduce disk I/O for retained segments.

Example configuration snippet (indented code block):

# postgresql.conf
max_replication_slots = 10
max_wal_size = 4GB
min_wal_size = 1GB
wal_keep_size = 512MB
max_slot_wal_keep_size = 2GB
wal_sender_timeout = 30s
wal_compression = on

Performance analysis should focus on two metrics: lag LSN (difference between pg_current_wal_lsn() and the slot's restart_lsn) and disk usage growth rate. A sustained lag above a few hundred megabytes usually indicates a bottleneck in the consumer's processing pipeline (e.g., synchronous network writes, heavy JSON serialization, or back‑pressure from downstream systems).

Slot Starvation, WAL Retention Overflow, and Data Loss Scenarios

Misconfiguration can lead to three dangerous states:

  • Slot starvation – when the server discards WAL before the slot catches up, the consumer receives a REPLICATION_ERROR and must restart from a new snapshot, potentially missing changes.
  • WAL retention overflow – if max_slot_wal_keep_size is too high and the disk cannot accommodate the retained segments, PostgreSQL may abort writes, causing a cluster halt.
  • Data loss – disabling wal_keep_size or setting wal_sender_timeout too aggressively can cause the slot to be considered inactive, leading to premature WAL recycling.

Mitigation strategies include:

  • Regularly monitor pg_replication_slots for restart_lsn drift.
  • Set alerts on disk usage thresholds.
  • Deploy a secondary consumer or a hot standby that can take over a stalled slot.
  • Use pg_replication_origin_advance to acknowledge progress only after durable storage of the change.

Key Takeaways

  • Logical decoding separates WAL parsing from output formatting, giving fine‑grained control over CDC pipelines.
  • Replication slots guarantee WAL retention but must be actively managed; feedback LSNs are the lifeline.
  • Plugin developers should keep snapshot scopes minimal to avoid unnecessary WAL buildup.
  • Tuning max_slot_wal_keep_size, wal_keep_size, and wal_sender_timeout balances latency against disk consumption.
  • Continuous monitoring of slot lag and disk growth prevents starvation and data‑loss scenarios.
