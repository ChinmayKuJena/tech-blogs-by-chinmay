Introduction

PostgreSQL logical replication slots act as durable bookmarks in the write‑ahead log (WAL). They guarantee that a consumer can resume from the exact point where it left off, even after a crash or a restart. While this durability is essential for change‑data‑capture pipelines, it also introduces a backpressure surface: the primary server must retain WAL records until every active slot has consumed them. Unchecked, this can lead to slot bloat, excessive disk usage, and latency spikes that cascade through downstream systems.

Understanding the backpressure path requires a view of three interacting components: the slot lifecycle in shared memory, the WAL read loop that feeds logical decoding plugins, and the checkpoint subsystem that decides when old WAL files can be recycled. The article walks through each component, shows the relevant source snippets, and provides production‑grade tuning guidance.

Slot Lifecycle and Backpressure Signaling

A logical replication slot is created with pg_create_logical_replication_slot. Internally PostgreSQL allocates a LogicalDecodingContext and a ReplicationSlot structure stored in the ReplicationSlots shared memory array. The slot transitions through the following states:

  1. Active – a client holds a replication connection and periodically calls pg_logical_slot_get_changes. The slot’s restart_lsn is updated to the latest WAL position that has been safely decoded.
  2. Inactive – the client disconnects; the slot remains in the catalog but its active flag is cleared. The restart_lsn stops advancing, causing the primary to retain WAL.
  3. Dropped – pg_drop_replication_slot removes the slot entry and frees its resources.

Backpressure is signaled through the restart_lsn. The primary’s WAL recycling logic checks the minimum restart_lsn across all slots. If the minimum lsn lags behind the current WAL insert point, the system must keep the intervening WAL segments, inflating pg_wal size.

Shared Memory Structures and Persistence

The core structures live in src/backend/access/transam/replication.c.

// Simplified view of the slot structure
typedef struct ReplicationSlot
{
    char        name[NAMEDATALEN];
    Oid         database;
    bool        active;
    XLogRecPtr  restart_lsn;   // Backpressure marker
    XLogRecPtr  confirmed_flush;
    bool        is_logical;
    /* ... other fields ... */
} ReplicationSlot;

Slots are persisted to pg_replication_slots on checkpoint. The slot_save routine writes the in‑memory array to disk, ensuring that a crash does not lose the restart_lsn values. Because persistence occurs only at checkpoint, a sudden surge of inactive slots can delay the release of WAL until the next checkpoint, amplifying backpressure.

WAL Read Loop and Logical Decoding

Logical decoding plugins implement a callback that receives a stream of XLogRecord structures. The generic loop in src/backend/replication/logical/logical.c looks like:

while (!slot->active)
    WaitLatch(...);
while (true)
{
    XLogRecPtr start = GetXLogInsertRecPtr();
    XLogRecord *record = XLogReadRecord(start, LOGICAL);
    if (!record)
        break; // reached end of WAL
    DecodeRecord(context, record);
    slot->restart_lsn = record->EndRecPtr; // advance backpressure marker
}

The loop advances restart_lsn only after a record has been successfully decoded and handed to the client. If the client processes records slowly, the loop stalls, and the primary retains WAL.

Performance and Scaling Analysis

WAL Generation Rate vs. Slot Count

The retention cost grows linearly with the lag between the fastest and slowest slot. In a workload generating 200 MB/s of WAL, a single lagging slot that falls behind by 10 seconds forces PostgreSQL to keep an extra 2 GB of WAL files. Multiplying this by dozens of slots can exhaust pg_wal storage.

Disk I/O Impact

When WAL cannot be recycled, the system continues to allocate new segment files, leading to higher I/O pressure on the storage subsystem. Monitoring pg_stat_bgwriter for wal_buffers_full and wal_writer_delay spikes can reveal backpressure‑induced I/O bottlenecks.

Slot Count Limits

The max_replication_slots GUC caps the number of slots. Exceeding this limit triggers ERROR: could not allocate replication slot. However, the limit does not protect against bloat; a single inactive slot can consume the same resources as many active ones.

Metrics Collection

Key metrics to watch:

  • pg_replication_slots view – active, restart_lsn, confirmed_flush
  • pg_stat_replication – write_lag, flush_lag
  • pg_stat_wal – wal_size, wal_files
  • pg_stat_bgwriter – wal_buffers_full

Collecting these metrics at a 10‑second interval provides enough granularity to spot emerging backpressure before it impacts latency.

Common Failure Mode: Slot Bloat

When a consumer crashes or is deliberately stopped, its slot remains in the catalog with a stale restart_lsn. The primary cannot truncate WAL, and the pg_wal directory grows unchecked. Over time, this leads to:

  • Disk exhaustion on the primary
  • Increased checkpoint duration
  • Elevated replication lag for other slots sharing the same WAL stream

Mitigation Strategies

  1. Idle Slot Timeout – Use wal_receiver_timeout for physical replication; for logical slots, implement an external watchdog that drops slots idle beyond a configurable threshold.
  2. Graceful Shutdown Hook – Ensure applications call pg_logical_slot_peek_changes with skip_empty_xacts before disconnecting, allowing the slot to advance to the latest LSN.
  3. Slot Reuse – Instead of creating a new slot per job, reuse a long‑lived slot and track consumer offsets in a separate table.
  4. Periodic Vacuum of pg_replication_slots – A scheduled job that queries pg_replication_slots for inactive slots older than N hours and drops them after verification.

Tuning Recommendations

Parameter Typical Range Effect
max_replication_slots 10‑100 Upper bound on slot objects; increase only if needed.
wal_keep_size 0‑5 GB Guarantees a minimum WAL reserve; keep low to avoid masking backpressure.
checkpoint_timeout 5‑15 min Shorter intervals free WAL sooner but increase checkpoint overhead.
wal_receiver_status_interval 10‑30 s Controls how often replicas report progress; tighter intervals improve backpressure detection.
logical_decoding_work_mem 64‑256 MB Limits memory used per decoding session; prevents OOM on large transactions.

Adjust these settings in concert with application‑level flow control. For example, coupling a consumer‑side queue size limit with a server‑side wal_receiver_status_interval of 15 seconds provides a predictable backpressure window.

Trade‑offs and Pitfalls

  • Aggressive Checkpointing – Reducing checkpoint_timeout frees WAL faster but can degrade overall throughput due to more frequent I/O syncs.
  • Large wal_keep_size – Masks slot lag, making the problem harder to detect; should be used only for short‑term maintenance windows.
  • Automatic Slot Dropping – Simplifies operations but risks data loss if a consumer is temporarily unavailable.
  • Indeterminate Consumer Lag – Logical decoding plugins that batch records may report progress only after a batch completes, artificially inflating perceived lag.

Balancing these factors requires a clear SLA for replication latency and a monitoring pipeline that alerts on the metrics listed earlier.

Key Takeaways

  • Logical replication slots retain WAL until every active slot advances its restart_lsn; this is the core backpressure mechanism.
  • Slot lifecycle, shared memory structures, and checkpoint coordination together determine how quickly WAL can be recycled.
  • Slot bloat arises from inactive slots that hold stale LSNs; proactive cleanup and graceful shutdown hooks are essential.
  • Monitoring pg_replication_slots, pg_stat_wal, and checkpoint metrics provides early warning of backpressure buildup.
  • Production tuning involves setting sensible limits on slot count, checkpoint frequency, and WAL retention while aligning consumer flow control with server‑side signaling.

By treating replication slots as a bounded resource rather than an infinite queue, operators can maintain predictable latency, avoid disk‑space emergencies, and keep logical replication pipelines performant at scale.
