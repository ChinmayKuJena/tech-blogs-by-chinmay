Optimistic vs Pessimistic Locking: Picking the Right Concurrency Control

📅 2026-04-02
A quick comparison of optimistic and pessimistic locking in relational databases, with Java/Spring examples to help you choose the right strategy for high‑contention workloads.
databases concurrency java spring-boot

Insight

When multiple requests try to update the same row, two main strategies exist. Optimistic locking assumes conflicts are rare; it checks a version column at commit time and throws an exception on mismatch. Pessimistic locking acquires a database lock up‑front, blocking other transactions until the current one finishes. Optimistic is lightweight and scales well under low contention, while pessimistic guarantees consistency under heavy write contention but can cause deadlocks and reduced throughput.

Example

// Optimistic locking with JPA @Version
@Entity
public class Order {
    @Id Long id;
    @Version Long version; // auto‑incremented on each update
    BigDecimal amount;
}

// Pessimistic lock in a repository method
public interface OrderRepo extends JpaRepository<Order, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select o from Order o where o.id = :id")
    Order findForUpdate(@Param("id") Long id);
}

Takeaway

Start with optimistic locking and only switch to pessimistic when you see frequent OptimisticLockException under load. Monitor lock wait times and deadlock counts to avoid hidden performance penalties.
