Idempotent Writes with PostgreSQL UPSERT (ON CONFLICT)

📅 2026-04-12
Learn how PostgreSQL's ON CONFLICT clause lets you turn ordinary INSERTs into safe, idempotent upserts, eliminating race conditions and duplicate‑key errors in high‑throughput APIs.
postgresql sql backend database

Insight

When an API endpoint must be safe to retry—e.g., after a network hiccup—traditional INSERT statements can cause duplicate‑key violations. PostgreSQL’s ON CONFLICT clause lets you express upsert logic in a single atomic statement, turning a retry into a no‑op or an update without extra application code.

Example

INSERT INTO users (email, name, created_at)
VALUES ('alice@example.com', 'Alice', now())
ON CONFLICT (email) DO UPDATE
  SET name = EXCLUDED.name,
      created_at = GREATEST(users.created_at, EXCLUDED.created_at);

The query inserts a new row or updates the existing one if the email already exists, preserving the most recent timestamp.

Takeaway

Prefer INSERT … ON CONFLICT for idempotent write paths. It removes the need for pre‑checks, reduces race conditions, and keeps your service stateless and retry‑friendly.
