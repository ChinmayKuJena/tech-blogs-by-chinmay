Optimistic Concurrency with Version Columns in TypeORM

📅 2026-04-10
Leverage a simple version column to prevent lost updates in high‑traffic services, using TypeORM's built‑in optimistic lock support.
typescript database concurrency undefined

Insight

When multiple API instances update the same row concurrently, the classic "last write wins" pattern silently overwrites data. Adding a tiny integer version column and letting the ORM enforce an optimistic lock turns this race into a deterministic error that you can handle gracefully. The database checks that the version you read matches the version at update time; any mismatch throws an exception, signalling a concurrent modification.

Example

import { Entity, PrimaryGeneratedColumn, Column, VersionColumn, Repository } from 'typeorm';

@Entity()
export class Account {
  @PrimaryGeneratedColumn()
  id: number;

  @Column('decimal')
  balance: number;

  @VersionColumn()
  version: number; // auto‑incremented on each UPDATE
}

async function transfer(repo: Repository<Account>, fromId: number, toId: number, amount: number) {
  const from = await repo.findOneOrFail({ where: { id: fromId } });
  const to = await repo.findOneOrFail({ where: { id: toId } });

  from.balance -= amount;
  to.balance += amount;

  // TypeORM will include the version in the WHERE clause
  await repo.save([from, to]);
}

If another request modifies either account between the findOne and save, TypeORM throws an OptimisticLockVersionMismatchError.

Takeaway

Add a @VersionColumn to critical tables and let the ORM surface concurrency conflicts as explicit errors. This cheap safeguard eliminates silent data loss without the overhead of heavyweight pessimistic locks.
