Optimistic Locking with TypeORM: Safe Concurrent Updates

📅 2026-04-20
Learn how a simple version column can prevent lost updates in high‑traffic services without heavyweight row locks.
backend typescript orm database

Insight

When multiple requests try to modify the same row, developers often reach for explicit SELECT ... FOR UPDATE locks, which can hurt throughput. In many CRUD workloads, an optimistic approach works better: store a version field, let the DB enforce that the version hasn't changed, and retry on conflict. This pattern avoids long‑lived locks while still guaranteeing data integrity.

Example

import { Entity, PrimaryGeneratedColumn, Column, VersionColumn } from 'typeorm';

@Entity()
export class Product {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  price: number;

  @VersionColumn()
  version: number; // auto‑incremented on each UPDATE
}

// Service layer
async function updatePrice(id: number, newPrice: number) {
  const repo = dataSource.getRepository(Product);
  const product = await repo.findOneByOrFail({ id });
  product.price = newPrice;
  try {
    await repo.save(product); // throws if version changed
  } catch (e) {
    // handle OptimisticLockVersionMismatchError – retry or abort
    throw new Error('Concurrent update detected');
  }
}

Takeaway

Add a @VersionColumn (or manual timestamp) to critical entities and let TypeORM raise a conflict when stale data is saved. A quick retry loop often resolves the race, giving you high concurrency without the performance penalty of pessimistic locks.
