Circuit Breaker with Resilience4j: Guarding All Flaky Dependencies

📅 2026-05-17
Learn why circuit breakers aren’t just for HTTP calls and how Resilience4j can protect any unreliable resource in a Spring Boot app.
java spring-boot resilience4j circuit-breaker

Insight

A common myth is that circuit breakers only make sense around outbound HTTP requests. In reality, any dependency that can fail or become slow—databases, message brokers, or even in‑process caches—benefits from a breaker. Resilience4j lets you wrap arbitrary Supplier or Callable logic, exposing metrics and fallback hooks without pulling in heavyweight libraries.

Example

@Bean
public CircuitBreakerRegistry circuitBreakerRegistry() {
    return CircuitBreakerRegistry.ofDefaults();
}

@Service
public class OrderService {
    private final CircuitBreaker cb;
    private final JdbcTemplate jdbc;

    public OrderService(CircuitBreakerRegistry registry, JdbcTemplate jdbc) {
        this.cb = registry.circuitBreaker("order-db");
        this.jdbc = jdbc;
    }

    public Order findById(Long id) {
        Supplier<Order> dbCall = () -> jdbc.queryForObject(
            "SELECT * FROM orders WHERE id = ?", new Object[]{id},
            (rs, rowNum) -> new Order(rs.getLong("id"), rs.getString("status")));
        return Try.ofSupplier(CircuitBreaker.decorateSupplier(cb, dbCall))
                 .recover(throwable -> new Order(id, "UNKNOWN"))
                 .get();
    }
}

The CircuitBreaker opens after a configurable failure threshold, instantly returning a fallback Order instead of hammering the DB.

Takeaway

Treat every external or resource‑intensive call as a potential failure point. Adding a lightweight Resilience4j circuit breaker around it protects your service’s latency and keeps downstream systems safe.
