Why More Indexes Can Hurt Performance: The Index Bloat Myth

📅 2026-04-19
Adding indexes feels safe, but each extra index consumes I/O, storage, and write time. Learn when an index is truly beneficial and how to avoid index bloat in PostgreSQL.
databases postgresql performance undefined

Insight

While indexes accelerate read queries, they also impose a hidden cost on every INSERT, UPDATE, and DELETE. In PostgreSQL, each additional B‑tree index adds extra page writes, increases table bloat, and can push the planner to choose sub‑optimal plans when statistics become stale. The common myth—more indexes = faster reads—breaks down once write latency and storage overhead outweigh the occasional read gain. Instead, focus on covering the most frequent query patterns, use partial or expression indexes, and regularly VACUUM ANALYZE to keep planner statistics accurate.

Example

-- Bad: blanket index on a high‑cardinality column
CREATE INDEX idx_users_email ON users(email);

-- Good: partial index used only for active users
CREATE INDEX idx_active_users_email ON users(email) WHERE status = 'active';

The partial index is smaller, stays in memory longer, and only affects rows that matter for the typical query SELECT * FROM users WHERE email = $1 AND status = 'active';.

Takeaway

Audit your indexes quarterly: drop unused ones, replace broad indexes with partial or covering variants, and monitor write latency after each change. A lean index set keeps both reads and writes performant.
