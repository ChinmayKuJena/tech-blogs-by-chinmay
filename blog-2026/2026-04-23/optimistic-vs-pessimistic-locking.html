Optimistic vs Pessimistic Locking: Picking the Right Concurrency Control

📅 2026-04-23
A quick guide to choosing optimistic or pessimistic locking in relational databases, with a TypeORM example.
databases concurrency typeorm postgresql

Insight

When multiple services update the same row, the choice between optimistic and pessimistic locking dramatically affects latency and contention. Optimistic locking assumes conflicts are rare; it stores a version column and checks it on update, letting the DB avoid heavy row locks. Pessimistic locking, by contrast, acquires a lock (SELECT … FOR UPDATE) at read time, guaranteeing exclusive access but increasing wait time and dead‑lock risk. Use optimistic locking for high‑throughput, low‑conflict workloads (e.g., user profile edits). Switch to pessimistic locking when the cost of a failed retry outweighs the overhead of holding a lock, such as inventory reservations.

Example

// TypeORM entity with version column for optimistic locking
@Entity()
export class Product {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  stock: number;

  @VersionColumn()
  version: number; // auto‑incremented on each update
}

// Updating stock with optimistic lock
await getRepository(Product).save({ id: 1, stock: 95, version: 3 });

Takeaway

Start with optimistic locking and add a version column; only fall back to SELECT … FOR UPDATE when business rules demand absolute consistency under heavy contention.
