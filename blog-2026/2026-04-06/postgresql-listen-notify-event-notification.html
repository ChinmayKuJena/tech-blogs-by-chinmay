Lightweight Event Notification with PostgreSQL LISTEN/NOTIFY

📅 2026-04-06
Use PostgreSQL's built‑in LISTEN/NOTIFY to push real‑time events to Node.js services without adding a separate message broker.
postgresql event-driven nodejs undefined

Insight

PostgreSQL’s LISTEN/NOTIFY provides a low‑overhead pub/sub mechanism that works perfectly for intra‑service events such as cache invalidation, audit logging, or simple workflow triggers. Because it’s native to the database, you avoid the operational cost of a dedicated broker and keep the event path short. The downside is that payload size is limited (max 8000 bytes) and it’s not meant for high‑throughput streaming, but for occasional signals it’s ideal.

Example

import { Client } from 'pg';

const client = new Client({ connectionString: process.env.DATABASE_URL });
await client.connect();
await client.query('LISTEN user_updated');
client.on('notification', msg => {
  const payload = JSON.parse(msg.payload!);
  console.log('User updated:', payload);
});

// Somewhere else in the app
await client.query('NOTIFY user_updated, $1', [JSON.stringify({ id: 42 })]);

Takeaway

When you need a simple, low‑latency signal between services that already share a PostgreSQL instance, LISTEN/NOTIFY is a pragmatic choice. Keep payloads tiny, monitor connection health, and reserve heavyweight brokers for high‑volume streams.
