Distributed Job Coordination with PostgreSQL Advisory Locks

📅 2026-04-17
Use PostgreSQL advisory locks to safely coordinate background jobs across multiple service instances without extra infrastructure.
postgresql advisory-locks distributed-systems sql

Insight

PostgreSQL provides lightweight, application‑level advisory locks that live outside transaction scope. They are perfect for coordinating cron jobs, batch workers, or leader election when you already have a DB connection pool and want to avoid a separate lock service. Unlike row‑level locks, advisory locks are identified by a 64‑bit key you choose, making them deterministic and easy to reason about. Because the lock is held until you explicitly release it (or the session ends), you can guarantee that only one instance of a job runs at a time, even across multiple pods or VM instances.

Example

import { Pool } from 'pg';
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

async function runSingletonJob() {
  const client = await pool.connect();
  try {
    // Try to acquire an exclusive advisory lock (key = 12345)
    const res = await client.query('SELECT pg_try_advisory_lock($1) AS acquired', [12345]);
    if (!res.rows[0].acquired) return console.log('Another instance is running');
    // ...perform job logic here...
    console.log('Job started');
    // simulate work
    await new Promise(r => setTimeout(r, 5000));
  } finally {
    // Release the lock regardless of success/failure
    await client.query('SELECT pg_advisory_unlock($1)', [12345]);
    client.release();
  }
}
runSingletonJob();

Takeaway

When you already have PostgreSQL in your stack, leverage its advisory locks for cheap, reliable job coordination—no extra Redis or Zookeeper needed. Just pick a stable key, acquire with pg_try_advisory_lock, and always release in a finally block.
