Avoiding Connection Pool Exhaustion in Serverless Backends

📅 2026-03-25
Why naive DB pooling breaks in serverless functions and how to keep connections healthy with lazy initialization and explicit cleanup.
databases serverless typescript performance

Insight

Serverless platforms spin up a fresh container per request, but they also reuse containers for subsequent invocations. If you create a long‑lived pg.Pool at module load time, each container can retain dozens of idle connections. When many containers are warm, the total connections quickly exceed the database limit, causing throttling or crashes. The safe pattern is to lazily instantiate a pool per request and close it before the function exits, or use a short‑lived client for one‑off queries.

Example

import { Client } from 'pg';

export const handler = async (event: any) => {
  const client = new Client({ connectionString: process.env.DATABASE_URL });
  await client.connect();
  const res = await client.query('SELECT now()');
  await client.end(); // ensures the socket is released
  return { statusCode: 200, body: JSON.stringify(res.rows) };
};

Takeaway

Never keep a global connection pool in a serverless function. Open a client, run your query, and close it explicitly to keep the total connection count predictable and within your DB limits.
