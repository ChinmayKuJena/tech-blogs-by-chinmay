Exponential Backoff with Jitter: Safer Retries for Unreliable APIs

📅 2026-05-05
Learn why plain exponential backoff can still cause thundering‑herd problems and how adding jitter creates a more resilient retry strategy for external service calls.
backend typescript retries reliability

Insight

When a service repeatedly fails, a naïve exponential backoff (e.g., 1s, 2s, 4s…) can still cause many clients to retry at the same moment, overwhelming the downstream system—a classic thundering‑herd scenario. Adding a small random jitter to each delay spreads the retries over a window, reducing contention and giving the target service breathing room.

Example

function sleep(ms: number) {
  return new Promise(r => setTimeout(r, ms));
}

async function retry<T>(fn: () => Promise<T>, attempts = 5, base = 200) {
  for (let i = 0; i < attempts; i++) {
    try {
      return await fn();
    } catch (e) {
      const jitter = Math.random() * base; // 0‑base ms jitter
      const delay = Math.pow(2, i) * base + jitter;
      await sleep(delay);
    }
  }
  throw new Error('All retry attempts failed');
}

Takeaway

Combine exponential growth with a modest random jitter (e.g., 0‑200 ms) to make retry storms less likely. Keep the max backoff reasonable and always cap total attempts to avoid indefinite hangs.
