Coordinating Distributed Jobs with PostgreSQL Advisory Locks

📅 2026-03-20
Learn how PostgreSQL advisory locks can act as a lightweight, reliable coordination primitive for distributed background jobs without adding external services.
postgresql advisory-locks distributed-systems jobs

Insight

PostgreSQL advisory locks are session‑level mutexes that live inside the database. Because they are managed by the same transaction engine that handles your data, they avoid the split‑brain problem common with external lock services. Use them to ensure that only one worker processes a given job ID at a time, even across multiple instances.

Example

// TypeScript using node-postgres (pg)
import { Pool } from 'pg';
const pool = new Pool();

async function processJob(jobId: number) {
  const client = await pool.connect();
  try {
    // Try to acquire an exclusive advisory lock for this jobId
    const gotLock = await client.query('SELECT pg_try_advisory_xact_lock($1)', [jobId]);
    if (!gotLock.rows[0].pg_try_advisory_xact_lock) return; // another worker has it
    // Safe to run the job now
    await client.query('UPDATE jobs SET status = $1 WHERE id = $2', ['running', jobId]);
    // ... job logic ...
    await client.query('UPDATE jobs SET status = $1 WHERE id = $2', ['completed', jobId]);
  } finally {
    client.release();
  }
}

Takeaway

Use pg_try_advisory_xact_lock (or the session‑level variant) to serialize work on a per‑entity basis. It requires no extra infrastructure, rolls back automatically on transaction failure, and keeps your job coordination tightly coupled to the data it protects.
