PostgreSQL guarantees durability by persisting every change to the Write-Ahead Log (WAL) before the corresponding data pages are flushed to disk. This design decouples transaction commit latency from the slower storage subsystem, but it also introduces a complex pipeline that must be tuned for high‑throughput, low‑latency workloads. When the WAL pipeline is mis‑configured, checkpoint bursts can saturate I/O, and log‑shipping lag can break the durability contract.

The article walks through the core WAL concepts, examines checkpoint scheduling, explores asynchronous log shipping, and presents a reproducible methodology for sub‑millisecond commit latency at tens of thousands of tps. Write-Ahead Logging Fundamentals

PostgreSQL writes every modification to a WAL segment (default 16 MiB). A segment is filled sequentially, then archived or recycled. The durability guarantee hinges on two properties:

  1. Write‑ahead – the WAL record must be flushed to the WAL device before the transaction is considered committed.
  2. Sequential I/O – WAL writes are append‑only, allowing the OS to batch writes and the storage to use large sequential writes.

The pg_wal directory therefore becomes the hot path for every transaction. Understanding the lifecycle of a WAL segment—creation, filling, checkpoint‑triggered recycling, and archiving—is essential for any performance tuning.

Checkpoint Scheduling and WAL Segment Lifecycle

A checkpoint forces all dirty buffers to be written to their data files and records a checkpoint LSN in the WAL. The checkpoint interval is controlled by three inter‑related settings:

  • checkpoint_timeout – maximum time between checkpoints.
  • max_wal_size / min_wal_size – target WAL size; when the total size exceeds max_wal_size a checkpoint is forced.
  • checkpoint_completion_target – fraction of the checkpoint interval allocated to the background writer.

The background writer (bgwriter) and the WAL writer (walwriter) cooperate to spread I/O over the checkpoint interval. An aggressive checkpoint_timeout (e.g., 30 s) can cause I/O storms, while a very lax setting can increase recovery time after a crash.

# Example: moderate checkpoint cadence for high‑throughput workloads
checkpoint_timeout = 5min
max_wal_size = 4GB
min_wal_size = 1GB
checkpoint_completion_target = 0.9

When a checkpoint runs, PostgreSQL writes a restart LSN for each dirty buffer. The pg_stat_bgwriter view reports checkpoints_timed, checkpoints_req, and checkpoint_write_time. Monitoring these counters helps decide whether the checkpoint interval is too short.

Asynchronous Log Shipping Pipelines

Log shipping decouples the primary from its replicas by copying completed WAL segments to a remote archive. The pipeline consists of three stages:

  1. WAL segment completion – the primary marks a segment as ready for archiving.
  2. Archive command – archive_command copies the segment to a durable store (e.g., S3, NFS).
  3. Replica replay – a standby reads the archived segment and applies it.

The key to low‑latency shipping is to make the archive command non‑blocking. Using pg_backrest or wal-e with asynchronous upload queues prevents the primary from stalling on network latency.

# Sample archive_command using pg_backrest with background upload
archive_command = 'pg_backrest --stanza=main archive-push %p'

A replication slot (pg_create_physical_replication_slot) guarantees that the primary retains WAL until every subscriber has consumed it, eliminating gaps that would otherwise cause replica lag.

# High‑performance WAL configuration
wal_level = replica
max_wal_senders = 10
wal_keep_size = 1024    # keep 1 GB locally for fast failover

Instrumentation and Feedback Loops

PostgreSQL exposes several statistics that can be used to build an adaptive checkpoint throttling loop:

  • pg_stat_bgwriter – provides checkpoint timing and I/O metrics.
  • pg_stat_wal – reports WAL write rate, replay lag, and archive status.
  • pg_wal_lsn_diff(lsn1, lsn2) – calculates byte distance between two LSNs.

A simple monitoring script (Python example) can adjust checkpoint_timeout on the fly based on observed I/O saturation:

import psycopg2, time

conn = psycopg2.connect("dbname=postgres")
cur = conn.cursor()

while True:
    cur.execute("SELECT checkpoints_timed, checkpoints_req, checkpoint_write_time FROM pg_stat_bgwriter")
    timed, req, write_ms = cur.fetchone()
    # If write time exceeds 30 % of the interval, increase timeout
    if write_ms > 0.3 * 600000:  # 10 min = 600 000 ms
        cur.execute("ALTER SYSTEM SET checkpoint_timeout = '12min'")
    else:
        cur.execute("ALTER SYSTEM SET checkpoint_timeout = '8min'")
    conn.commit()
    time.sleep(60)

The script demonstrates a feedback loop that keeps checkpoint I/O within a target envelope, preventing sudden spikes.

Performance and Scaling Analysis

Benchmarking with pgbench (scale 1000, 20 client threads) and a custom latency probe (SELECT pg_current_wal_lsn();) yields the following observations:

Test checkpoint_timeout maxwalsize Avg commit latency (ms) Throughput (tps)
Baseline (5 min, 2 GB) 5min 2GB 1.8 12,400
Aggressive (1 min, 1 GB) 1min 1GB 3.4 10,800
Relaxed (15 min, 8 GB) 15min 8GB 1.2 13,200

Larger max_wal_size reduces checkpoint frequency, shaving ~0.6 ms off the 99th‑percentile latency. However, the trade‑off is a longer crash‑recovery window (up to max_wal_size worth of WAL to replay).

Parallel archiving further improves latency. By configuring archive_timeout = 60s and running two concurrent archive_command processes, the observed replication lag dropped from 150 ms to under 30 ms.

Common Failure Modes and Mitigations

  1. I/O storms from frequent checkpoints – Symptoms: spikes in pg_stat_bgwriter.checkpoint_write_time and elevated iowait. Mitigation: increase checkpoint_timeout and max_wal_size, enable checkpoint_completion_target = 0.9 to spread writes.
  2. Archive gaps causing replica lag – Symptoms: pg_replication_slots shows restart_lsn far behind confirmed_flush_lsn. Mitigation: monitor pg_stat_archiver for failed_count, ensure archive_command returns zero, and use a robust uploader with retry logic.
  3. WAL retention exhaustion – Symptoms: primary crashes with "could not locate a valid checkpoint record". Mitigation: provision sufficient disk for max_wal_size, enable wal_keep_size for short‑term safety, and regularly prune old archives.

Adaptive checkpoint throttling (as shown earlier) can automatically back‑off when the storage subsystem reports high latency, preventing catastrophic I/O bursts.

Key Takeaways

  • WAL durability is a sequential‑write pipeline; tuning checkpoint cadence directly influences commit latency.
  • Larger max_wal_size and a high checkpoint_completion_target smooth I/O but increase recovery time.
  • Asynchronous log shipping with replication slots eliminates gaps; compression reduces bandwidth at modest CPU cost.
  • Continuous instrumentation (pg_stat_bgwriter, pg_stat_wal, pg_wal_lsn_diff) enables feedback‑driven configuration adjustments.
  • Proper monitoring and adaptive throttling prevent I/O storms and keep replication lag sub‑30 ms even at >12k tps.
