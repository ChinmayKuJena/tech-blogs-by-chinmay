Using PostgreSQL LISTEN/NOTIFY for Lightweight Pub/Sub

📅 2026-05-02
Leverage PostgreSQL's built‑in LISTEN/NOTIFY to implement low‑volume event broadcasting without adding an external message broker.
postgresql pubsub database nodejs

Insight

PostgreSQL ships with a simple publish/subscribe mechanism via the LISTEN and NOTIFY commands. It’s ideal for low‑throughput, intra‑service events such as cache invalidations, audit logs, or feature‑flag flips. Because the broker lives inside your primary data store, you avoid extra infrastructure, latency, and operational overhead. The trade‑off is limited payload size (max 8000 bytes) and no guaranteed delivery semantics, so use it only for non‑critical signals.

Example

import { Client } from 'pg';

const client = new Client({ connectionString: process.env.DATABASE_URL });
await client.connect();

// Listener side
client.on('notification', msg => {
  console.log('Received event:', msg.payload);
});
await client.query('LISTEN new_user');

// Publisher side (could be another service)
await client.query("NOTIFY new_user, 'user_id=42'");

Takeaway

When you need a quick, zero‑dependency event channel, spin up a LISTEN/NOTIFY pair in PostgreSQL. Keep payloads tiny, treat messages as best‑effort, and monitor the pg_stat_activity view to ensure listeners stay healthy.
