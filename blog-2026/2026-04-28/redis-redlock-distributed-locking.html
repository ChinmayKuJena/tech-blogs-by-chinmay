Distributed Locking with Redis RedLock in TypeScript

📅 2026-04-28
Learn how to implement a safe distributed lock using Redis RedLock in a Node.js/TypeScript service to avoid race conditions across multiple instances.
redis typescript distributed-systems locking

Insight

When several stateless service instances need to coordinate access to a shared resource—such as a quota counter or a critical section—simple SETNX locks can break under network partitions or client crashes. Redis' RedLock algorithm mitigates these issues by acquiring the lock on multiple independent Redis nodes and requiring a majority consensus. This approach gives you a fault‑tolerant lock without adding a heavyweight coordination service like Zookeeper.

Example

import Redlock from 'redlock';
import IORedis from 'ioredis';

// Connect to three independent Redis nodes
const clients = [
  new IORedis({ host: 'redis-1.example.com', port: 6379 }),
  new IORedis({ host: 'redis-2.example.com', port: 6379 }),
  new IORedis({ host: 'redis-3.example.com', port: 6379 })
];

const redlock = new Redlock(clients, {
  // Retry for up to 2 seconds
  retryCount: 10,
  retryDelay: 200,
});

async function criticalSection() {
  const lock = await redlock.acquire(['locks:order:123'], 5000); // 5 s TTL
  try {
    // … safe work here …
  } finally {
    await lock.release();
  }
}

Takeaway

Use RedLock when you need a lightweight, highly available lock across horizontally scaled services. Keep the lock TTL short, handle retries gracefully, and always release the lock in a finally block to prevent deadlocks.
