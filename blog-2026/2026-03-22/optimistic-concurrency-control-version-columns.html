Optimistic Concurrency Control with Version Columns

📅 2026-03-22
Learn how a simple version column can protect your data from lost updates without heavyweight locks, and see a tiny TypeORM example that makes OCC painless.
backend database concurrency sql

Insight

Optimistic Concurrency Control (OCC) lets you avoid database row locks by detecting conflicts only at commit time. The common myth is that you need pessimistic locks for any concurrent write scenario, but a tiny version (or updated_at) column is often enough. Each update increments the version; if two transactions read the same version and try to write, the second fails, forcing a retry. This approach scales far better under read‑heavy workloads and works with any relational DB that supports atomic WHERE checks.

Example

-- Add a version column to an existing table
ALTER TABLE orders ADD COLUMN version INT NOT NULL DEFAULT 0;

// TypeORM update with OCC
await getRepository(Order).update(
  { id: orderId, version: currentVersion }, // WHERE clause includes version
  { status: 'shipped', version: () => 'version + 1' } // increment atomically
);

If the update returns 0 rows, another process has already modified the row and you should reload and retry.

Takeaway

Add a version (or timestamp) column to every write‑heavy table, and always include it in your WHERE clause when updating. A failed row‑count signals a conflict, letting you retry safely without ever acquiring explicit locks.
